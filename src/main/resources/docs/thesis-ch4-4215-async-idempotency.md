### 4.2.1.5 异步与幂等机制设计（补充类图 + 时序图）

本节从工程可靠性角度说明系统如何在高并发下实现“异步解耦 + 幂等防重 + 最终一致”。结合代码实现，本系统并非依赖单一幂等点，而是采用接入层、消费层、存储层三层协同机制：

- 接入层：入口请求按 `messageId`/`requestId` 去重；
- 消费层：Kafka 消费过程采用状态机迁移与手动确认；
- 存储层：依赖唯一键与乐观锁进行最终兜底。

#### （1）补充类图设计说明

如图4-19所示，异步与幂等机制涉及五类核心对象：

1. **接入控制对象**：`TaskEngineController` 负责请求受理、去重与同步/异步路由；
2. **编排与业务对象**：`TaskEngine`、`RiskDecisionServiceImpl`、`RewardServiceImpl`、`UserTaskInstanceServiceImpl` 负责任务执行与消息触发；
3. **消息对象**：`RewardProducer`、`RiskDecisionPersistProducer` 对应消息发布，`RewardConsumer`、`TaskPersistenceConsumer`、`RiskDecisionPersistConsumer`、`TaskConfigChangeConsumer` 对应消息消费；
4. **缓存与常量对象**：`RedisTemplate` 与常量类共同约束去重键、TTL 与主题名；
5. **存储对象**：`UserRewardRecord`、`RiskDecisionLog`、`UserTaskInstance` 提供数据库最终一致兜底。

该类图强调“职责分离”：控制器不承担复杂持久化逻辑，消费者不承担业务编排，幂等约束由多层机制共同保障。

#### （2）接入层异步与幂等机制

如图4-20所示，系统在事件入口首先基于 `messageId` 进行 Redis 去重。若命中重复消息，直接返回重复结果，不再进入业务处理。若 Redis 异常，则降级放行，优先保障接口可用性。

在请求标识方面，系统对 `requestId` 执行归一化处理：当长度超限时进行摘要化，避免后续写库字段超长。完成去重与标识处理后，进入两种调用路径：

- **同步路径**：在当前线程执行 `TaskEngine.processEvent`，返回即时处理结果；
- **异步路径**：任务提交线程池，接口提前返回 `202`；若线程池拒绝，则返回 `503`，避免无限排队拖垮系统。

这一路径的核心设计要点是：入口快速失败、可降级、可追踪。

#### （3）奖励与风控异步消费幂等机制

如图4-21所示，系统存在两条关键异步消费链路。

**1）奖励发放链路（reward-topic）**

奖励消息包含 `messageId`，消费者先“初始化记录 + 抢占处理状态”，再执行发放策略。状态迁移约束为：

- `INIT/FAILED -> PROCESSING`；
- `PROCESSING -> SUCCESS`；
- 异常时 `INIT/PROCESSING -> FAILED`。

若消息已成功处理或处于处理中，消费者直接跳过，从而避免重复发放。发放成功后再写入 Redis 去重标记，形成“状态机幂等 + 缓存幂等”双保险。

**2）风控落库链路（risk-decision-persist-topic）**

风控主流程将决策日志异步发送到 Kafka，消费者落库时先做 Redis 去重，随后写入 `risk_decision_log`。若出现 `request_id` 唯一键冲突，按幂等成功处理并确认消费；若为其他异常，则发送 DLQ 并抛出异常参与重试。

这保证了风控主链路与审计落库解耦，同时避免重复消息导致日志膨胀。

#### （4）配置变更异步刷新机制

如图4-22所示，`task_config` 表变更通过 Binlog 被 Canal 捕获，写入 Kafka 后由 `TaskConfigChangeConsumer` 处理。消费者仅处理 `task_config` 事件，并对每个任务执行“失效本地缓存 + 刷新缓存”。

该链路采用 `MANUAL_IMMEDIATE` 手动确认策略：

- 非目标表消息直接 ack 忽略；
- 单条处理异常记录日志并 ack，防止毒丸消息阻塞分区；
- 使用实例级 groupId，确保每个应用实例都能接收刷新事件。

其设计目标是“配置变更可快速传播，且不因单条异常导致整体停滞”。

#### （5）幂等机制小结

本系统幂等机制可概括为“三层协同”：

1. **入口防重**：`messageId/requestId` 先挡住重复请求；
2. **消费防重**：状态迁移约束 + 去重键，防止重复执行副作用；
3. **存储兜底**：唯一键与乐观锁保障最终一致。

通过以上机制，系统在高并发与异步重试场景下，能够在保证吞吐的同时控制重复处理风险，满足任务激励业务对一致性与稳定性的要求。
